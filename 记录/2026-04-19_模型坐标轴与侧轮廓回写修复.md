## 一、改动背景

本次开发在 2026-04-18「模型旋转替代相机旋转、Z轴俯视视角重构」基础上，继续修复和完善以下问题：

1. 第一个三维窗口缺少可视化坐标轴，难以判断模型局部朝向
2. 尖/根侧轮廓点在保存、上传、重新读取后，无法稳定回写并恢复
3. 尖侧轮廓中的 `D` 点位置不稳定，未能保持在 `D_P1_J` 与 `D_P1_I` 中间
4. 旋转滑条在拖动时会从单位四元数姿态开始转，而不是从后端初始四元数姿态开始
5. 上传结果保存的四元数再次读取时，与前端滑条实时显示的姿态不一致
6. 尖侧轮廓窗口在某些刷新路径下会出现两个 `D` 点

本次目标是把「模型轴显示」「侧轮廓点 round-trip」「滑条旋转一致性」「尖侧轮廓渲染稳定性」这几条链路一起补齐。

## 二、核心改动

### 1. 新增模型局部坐标轴与 X/Y/Z 标签 (`ThreeScene.vue`)

**初始实现：**
- 在 `onMounted` 初始化 Three.js 场景时加入 `AxesHelper`
- 坐标轴最终挂载到 `modelGroup`，使其跟随模型一起旋转

**当前结果：**
```ts
const axesHelper = new THREE.AxesHelper(100);
modelGroup.add(axesHelper);
```

同时新增了轴标签：
- `X`：红色
- `Y`：绿色
- `Z`：蓝色

标签同样加入 `modelGroup`，因此会随着模型一起转动，作为模型局部坐标系显示。

### 2. 侧轮廓点加载/保存链路修复 (`App.vue`)

#### 2.1 新增侧轮廓点提取函数
为保证后端 `outputData` 能正确回填到前端共享状态，新增：

- `extractSideProfilePoints(outputData)`

提取点包括：
- 根侧：`G`, `H`, `A_P1_G`, `A_P1_H`
- 尖侧：`I`, `J`, `D_P1_I`, `D_P1_J`

#### 2.2 修正加载时的共享状态恢复
在以下加载路径中补充恢复 `sharedPointsState.sideProfilePoints`：

- `fetchNextDataBatch()`
- `loadDataByGroupId()`
- localStorage 回退恢复

#### 2.3 localStorage 持久化补充
在写入 localStorage 的对象中增加：

```ts
sideProfilePoints: sharedPointsState.sideProfilePoints
```

这样页面刷新、后端失败回退、切场景恢复时，不会丢失侧轮廓点。

#### 2.4 分类逻辑修正
在 `transformOutputData()` 中，将侧轮廓点从 `bezierPoints` 分类中排除，避免其错误混入主曲线控制点集合。

### 3. 侧轮廓面板改为从正确的数据源回填 (`ThreeScene.vue`)

之前 `ThreeScene.vue` 中侧轮廓面板的数据同步主要依赖 `sharedPointsState.bezierPoints`，这会导致：

- 后端原始侧轮廓点虽然已经加载进 `sideProfilePoints`
- 但根/尖侧轮廓视图并不真正使用这份数据

本次将同步函数重写为：

- 根/尖侧普通侧轮廓点优先取 `sharedPointsState.sideProfilePoints`
- `A` / `D` 等主锚点优先取 `sharedPointsState.positions`
- 必要时兼容回退到 `bezierPoints`

新增函数：
- `syncSideProfilePointsFromStore()`

并监听：
- `sharedPointsState.sideProfilePoints`
- `sharedPointsState.positions`
- `sharedPointsState.bezierPoints`

从而保证加载后 UI 能正确显示原始侧轮廓点。

### 4. 侧轮廓编辑写回修复 (`ThreeScene.vue`)

#### 根侧轮廓
- `A` 写回 `sharedPointsState.positions`
- 其余点写回 `sharedPointsState.sideProfilePoints`

#### 尖侧轮廓
- `D` 写回 `sharedPointsState.positions`
- 其余点写回 `sharedPointsState.sideProfilePoints`

这样保证：
- 根/尖侧轮廓窗口中的编辑
- 保存 JSON
- 上传结果
- 后端再次读取

都走同一份共享状态，而不会丢失。

### 5. 尖侧轮廓 `D` 点位置约束修复 (`ThreeScene.vue`)

开发中发现：
- 尖侧轮廓中的 `D` 点直接取模型主点 `D` 时，位置可能不符合尖侧轮廓的几何直觉
- 尖侧轮廓使用的是局部坐标系（x=宽度方向），D 应为原点 (0,0,0)

**2026-04-20 更新**：D 点已改为固定值 `{ x: 0, y: 0, z: 0 }`，不再取模型主坐标系中的 D 点坐标，也不再使用控制点中点计算。

同时添加了 D 点写回防护，防止尖侧轮廓中的 (0,0,0) 泄漏到主模型坐标：

```ts
if (sharedPositionPoint && (newPosition.x !== 0 || newPosition.y !== 0 || newPosition.z !== 0)) {
  // 才写回主模型
}
```

详见 `记录/2026-04-20_滑条增量旋转与预设视角修复.md` 第七、九节。

### 6. 旋转滑条与保存四元数逻辑统一 (`ThreeScene.vue`, `App.vue`)

这是本次最关键的一组修复。

#### 6.1 修复滑条从单位四元数开始旋转的问题
此前 `applySlidersToCamera()` 中，虽然计算了 `baseQuaternion`，但最后仅把滑条增量四元数赋给模型：

```ts
modelGroup.quaternion.copy(deltaQuaternion);
```

这会导致：
- 只要滑条一动
- 模型就像从 `1,0,0,0` 开始旋转
- 丢失后端提供的初始姿态

#### 6.2 初版统一：提取统一的四元数组合函数
分别在 `ThreeScene.vue` 与 `App.vue` 中新增：

- `composeQuaternionFromBaseAndDeltas(baseQuaternion, pitchDeg, yawDeg, rollDeg)`

初版逻辑为：
1. 取基准姿态下旋转后的局部轴
2. 分别构建 `pitchQ` / `yawQ` / `rollQ`
3. 使用统一顺序：
   ```ts
   rollQ.clone().multiply(yawQ).multiply(pitchQ)
   ```
4. 最终返回：
   ```ts
   deltaQ.multiply(baseQuaternion.clone())
   ```

#### 6.3 初版统一后的收益
- `ThreeScene.vue` 的 `applySlidersToCamera()` 改为调用统一函数
- `App.vue` 的 `getCorrectedQuaternion()` 也改为调用同一函数

这样先修复了两个问题：
1. 滑条从后端初始四元数姿态开始微调
2. 上传结果保存的 quaternion 再次读取时，和前端实时看到的姿态一致

#### 6.4 二次修正：改为严格绕模型自身局部轴旋转
后续联调中又发现一个问题：
- `yaw` / `pitch` 看起来是分别绕模型自身 `y/x` 轴旋转
- 但 `roll` 的表现不完全等同于绕模型自身 `z` 轴
- 在某些姿态下会出现 `roll` 与 `pitch` 调节耦合甚至重合的现象

根因是初版实现虽然使用了“基准姿态旋转后的轴”，但组合方式仍会让 `roll` 的表现不够严格地等同于局部 `z` 轴滚转。

因此又将 `composeQuaternionFromBaseAndDeltas()` 重写为：

```ts
const pitchLocalQ = new THREE.Quaternion().setFromAxisAngle(
  new THREE.Vector3(1, 0, 0), pitchRad
);
const yawLocalQ = new THREE.Quaternion().setFromAxisAngle(
  new THREE.Vector3(0, 1, 0), yawRad
);
const rollLocalQ = new THREE.Quaternion().setFromAxisAngle(
  new THREE.Vector3(0, 0, 1), rollRad
);

const localDelta = pitchLocalQ.clone().multiply(yawLocalQ).multiply(rollLocalQ);
return baseQuaternion.clone().multiply(localDelta);
```

现在三条滑条的语义统一为：
- `pitch`：绕模型自身局部 `x` 轴
- `yaw`：绕模型自身局部 `y` 轴
- `roll`：绕模型自身局部 `z` 轴

并且这套局部轴组合逻辑仍同时用于：
- 前端实时显示
- 上传结果时的 quaternion 保存

从而继续保证显示与保存的一致性。

### 7. 尖侧轮廓窗口重复 `D` 点修复 (`TipSideProfile.vue`)

之前 `TipSideProfile.vue` 的 `updateTipProfile()` 在清理旧对象时只移除：
- `draggableObjects`
- `pointLabels`
- 曲线与基线

但 `D` 点本身不参与拖拽，因此并不在 `draggableObjects` 中，这就导致：
- 每次刷新窗口时旧的 `D` 没被清掉
- 新的 `D` 又被创建一次
- 偶发出现两个 `D` 点

#### 修复方式
新增：
```ts
const pointMeshes: THREE.Mesh[] = []
```

在 `updateTipProfile()` 开头：
- 统一移除 `pointMeshes`
- 清理 `tipProfileCurve` / `baseLine`
- 同时执行 geometry/material dispose
- 清空数组并将对象置 null

在创建每个点时：
```ts
pointMeshes.push(sphere)
```

这样无论点是否可拖拽，刷新时都会被正确清理。

## 三、最终效果

本轮改动完成后，系统具备以下能力：

1. 第一个三维窗口中显示模型局部坐标轴及 `X/Y/Z` 标签
2. 坐标轴跟随模型旋转，便于观察模型本地坐标系
3. 根侧/尖侧轮廓点可以在保存、上传、再次读取后稳定回写
4. 尖侧轮廓中的 `D` 点固定为局部坐标系原点 `(0, 0, 0)`（2026-04-20 更新：不再使用控制点中点）
5. 滑条从后端原始四元数姿态开始旋转，而不是从单位四元数开始
6. 上传结果保存的 quaternion 与前端实时显示姿态一致
7. `pitch / yaw / roll` 统一为绕模型自身局部 `x / y / z` 轴旋转
8. 尖侧轮廓窗口中的 `D` 点不再偶发重复显示
