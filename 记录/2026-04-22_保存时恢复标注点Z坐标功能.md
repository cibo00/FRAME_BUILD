# 2026-04-22 保存时恢复标注点 Z 坐标功能

## 背景

`nailmodel.py` 中的 `get_input_points` 函数具备"恢复标注点 Z 坐标"的能力：对 `ZERO_Z_COMPAT_POINT_NAMES` 中的 18 个点，通过在填充曲面上做 XY 垂直投影采样 Z 值，加上 A 点原始 Z 偏移量，恢复到原始坐标系的 Z。

但前端代码中完全没有对应实现——`buildOutputData()` 保存时将所有点序列化为 `[x, y]`，Z 坐标被丢弃。

## 涉及的 18 个点名（与 nailmodel.py ZERO_Z_COMPAT_POINT_NAMES 一致）

```
A, B, C, D, E, F,
A_P_AF, D_P_CD, B_P_AB, F_P_AF,
AB_P2, AB_P3, BC_P2, CD_P1,
AF_P2, AF_P3, FE_P2, ED_P1
```

## 实现方案

### 保存时：前端通过曲面投影恢复 Z

仿照 `regularizeEdgeZ` 的最近邻逻辑，在保存时对 18 个点做 XY 最近邻采样，从曲面点中获取 Z 值，输出 `[x, y, recoveredZ]`。

### 读取时：前端 + Go 后端双重置零

保存文件带 `[x, y, z]`，但再次读取时需要像原数据一样忽略 Z：
- **前端** `transformOutputData()` 对 ZERO_Z_COMPAT 点强制 `z = 0`
- **Go 后端** `generateData.go` 对 ZERO_Z_COMPAT 点截断为 `[x, y]`

双重置零是防御性设计，确保任一端都能正确处理。

### Python nailmodel.py 不修改

`_expand_input_point` 中对 `ZERO_Z_COMPAT_POINT_NAMES` 的 2 元素输入已自动置零 Z，3 元素输入会直接使用 Z 值。由于 Go 后端和前端已在读取时截断/置零，Python 端收到的仍然是 `[x, y]` 格式，无需改动。

## 修改文件清单

### 前端项目 (Frame_Build)

#### 1. `src/stores/sharedStore.ts`
- `sharedPointsState` 新增 `surfacePoints: { x: number, y: number, z: number }[]` 字段（默认空数组）
- 用途：ThreeScene 计算完曲面后存入，App.vue 保存时读取

#### 2. `src/components/ThreeScene.vue`
- `generateCustomBezierCurves` 函数末尾（`allMeshPointsForExport.value = toSamplePoints` 之后）新增一行：
  ```typescript
  sharedPointsState.surfacePoints = toSamplePoints.map(p => ({ x: p.x, y: p.y, z: p.z }));
  ```

#### 3. `src/App.vue`
- 新增模块级常量 `ZERO_Z_COMPAT_POINT_NAMES`（Set，18 个点名）
- 新增 `recoverZFromSurface(x, y, surfacePoints)` 函数：XY 最近邻采样，返回最近曲面点的 z 值
- `buildOutputData()`：对 ZERO_Z_COMPAT 点输出 `[x, y, recoveredZ]`，其余点保持 `[x, y]`
- `transformOutputData()`：读取时对 ZERO_Z_COMPAT 点强制 `z = 0`

### Go 后端项目 (Go_Build)

#### 4. `generateData.go`
- 新增 `stripZForCompatPoints(od *OutputData)` 函数：对 18 个 ZERO_Z_COMPAT 字段，如果长度 > 2 则截断为前 2 个元素
- 在 `initializeData()` 中 `mergedOutputData` 组装完成后调用 `stripZForCompatPoints(&mergedOutputData)`

## 数据流

```
保存：前端 3D 场景 → buildOutputData() → 曲面投影恢复Z → [x, y, z] → JSON文件
读取：JSON文件 → Go后端 stripZ → [x, y] → 前端 transformOutputData z=0 → 3D场景
```
