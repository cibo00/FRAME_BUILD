- `src/components/ThreeScene.vue:425`

- `src/components/ThreeScene.vue:430`

核心代码逻辑：

```ts
const pitchLocalQ = new THREE.Quaternion().setFromAxisAngle(
  new THREE.Vector3(1, 0, 0), THREE.MathUtils.degToRad(pitchDeg)
);
const yawLocalQ = new THREE.Quaternion().setFromAxisAngle(
  new THREE.Vector3(0, 1, 0), THREE.MathUtils.degToRad(yawDeg)
);
const rollLocalQ = new THREE.Quaternion().setFromAxisAngle(
  new THREE.Vector3(0, 0, 1), THREE.MathUtils.degToRad(rollDeg)
);

const localDelta = pitchLocalQ.clone().multiply(yawLocalQ).multiply(rollLocalQ);
return baseQuaternion.clone().multiply(localDelta);
```

#### 当前确认后的基础公式

- `pitch`：绕局部 X 轴旋转
- `yaw`：绕局部 Y 轴旋转
- `roll`：绕局部 Z 轴旋转
- 角度转弧度：`rad = deg * π / 180`
- 最终目标四元数：

```text
Q_target = Q_base * Q_pitch * Q_yaw * Q_roll
```

这里的 `Q_base` 已不再简单等同于“基础欧拉角转四元数”，而是统一改为基于当前场景原始 `rotation` 构造：

```text
Q_base = buildFrontendQuaternionFromRotation(scene.rotation).quaternion
```

也就是：

- 若场景原始数据是 quaternion，则先走 quaternion 主链路
- 若场景原始数据是 Euler，则走 Euler 兼容链路
- 滑条增量始终叠加到“当前场景自己的原始姿态”上，而不是叠加到跨场景共享状态上

应用位置：

- `src/components/ThreeScene.vue:433`
- `src/components/ThreeScene.vue:442`
- `src/components/ThreeScene.vue:450`

```ts
const baseQuaternion = buildFrontendQuaternionFromRotation(rotation).quaternion;
const targetQuaternion = composeQuaternionFromBaseAndDeltas(...);
modelGroup.quaternion.copy(targetQuaternion);
```

#### 说明

虽然界面文案是“摄像机角度调整”，但实际被赋值的是：

- `src/components/ThreeScene.vue:450`

```ts
modelGroup.quaternion.copy(targetQuaternion);
```

因此这部分本质上是模型组姿态旋转，而不是直接修改 `camera` 对象本身。

这也是当前 quaternion 修复的关键前提：

- 后端存的是 `cameraRotation`
- 前端显示时实际转的是 `modelGroup`
- 因此前端显示姿态必须使用“后端相机四元数的逆”

此前多场景间出现“一个场景调滑条、另一个场景也像是被影响”的现象，根因不是滑条增量映射串场景，而是：

- 多个场景原始 `cameraRotation` 相同的时候
- 切场景后 `modelGroup.quaternion` 没有立即按当前场景自己的原始姿态重算
- 导致视觉上沿用了上一个场景的姿态

当前修复后：

- 每个场景滑条增量仍单独保存
- 每次切场景都会先应用该场景原始 `rotation`
- 再立即叠加该场景自己的 `pitch / yaw / roll` 增量

因此现在应满足：

```text
SceneDisplayQuaternion(scene_i)
= Q_base(scene_i) * Q_pitch(scene_i) * Q_yaw(scene_i) * Q_roll(scene_i)
```

不同场景之间不再共享显示结果。

## 4. 后端旋转恢复公式

位置：

- `src/components/ThreeScene.vue:455`
- `src/components/ThreeScene.vue:462`
- `src/components/ThreeScene.vue:476`
- `src/components/ThreeScene.vue:482`
- `src/components/ThreeScene.vue:498`

本项目支持两类后端 `rotation`：

### 4.1 后端直接给 quaternion

这部分是本次修复里最重要的协议确认点。

当前已经确认：

- 后端 quaternion 协议是 `[w, x, y, z]`
- three.js `THREE.Quaternion` 构造顺序是 `(x, y, z, w)`
- 后端字段语义是 `cameraRotation`
- 前端实际作用对象是 `modelGroup.quaternion`

所以当前正确显示链路不是“直接把后端四元数塞给 three.js”，而是：

```text
Q_backend(camera, [w,x,y,z])
-> reorder to Three.js [x,y,z,w]
-> normalize
-> Q_model = inverse(Q_backend)
```

对应实现可概括为：

```ts
const qBackend = new THREE.Quaternion(
  rotation[1],
  rotation[2],
  rotation[3],
  rotation[0],
).normalize();

const qModel = qBackend.clone().invert();
targetQuaternion = qModel;
```

即公式：

```text
Q_backend_three = Quaternion(x=rx, y=ry, z=rz, w=rw)
  where [rw, rx, ry, rz] = backendRotation

Q_model = inverse(Q_backend_three)
```

#### 为什么必须这样处理

用户实验曾将多个场景后端四元数统一改成 `1,0,0,0`。

如果把它当作 three.js 的 `[x,y,z,w]` 解读，则等价于：

```text
Quaternion(1,0,0,0)
```

这不是单位四元数，而是绕 X 轴旋转 180°。

而按后端协议 `[w,x,y,z]` 解读时：

```text
[w,x,y,z] = [1,0,0,0]
=> three.js quaternion = [0,0,0,1]
```

这才是单位四元数。

因此这次排查确认：此前 quaternion 主问题的根因就是协议顺序误判。

### 4.2 后端给 Euler 角

Euler 路径目前仍保留兼容逻辑：

```ts
x_from_backend = -x_from_backend;
y_from_backend = -y_from_backend;
z_from_backend = 90 - z_from_backend;
```

再转为 three.js Euler：

```ts
const euler = new THREE.Euler(
  THREE.MathUtils.degToRad(x_from_backend),
  THREE.MathUtils.degToRad(y_from_backend),
  THREE.MathUtils.degToRad(z_from_backend),
  'XYZ'
);
targetQuaternion = new THREE.Quaternion().setFromEuler(euler);
```

对应公式：

```text
x' = -x
y' = -y
z' = 90 - z
Q = Quaternion.fromEuler(x', y', z', 'XYZ')
```

#### 现阶段判断

这条 Euler 兼容链路当前没有继续大改，原因是：

- 当前 quaternion 主链路已经稳定
- Euler 路径中的 `-x / -y / (90 - z)` 明显带有历史补偿痕迹
- 在没有额外回归验证前，不宜继续改动以免破坏现有行为

因此当前文档中的结论是：

- quaternion 主链路：协议已明确，当前公式可信
- Euler 兼容链路：仍可用，但属于后续需要进一步审查的风险点

### 6.7 显式 sin / cos 几何旋转相关公式

位置：

- `src/components/ThreeScene.vue:1959` 附近

关键代码：

```ts
const alpha = Math.max(1e-4, Math.min(Math.PI / 2 - 1e-4, aTangentValue.value));
const phi_c = Math.max(1e-4, Math.min(Math.PI - 1e-4, Math.abs(AD_arc)));
const span = AD.length();
const adUnit = AD.clone().normalize();
const denom = Math.cos(alpha) - Math.cos(alpha + phi_c);

if (Math.abs(denom) > 1e-8 && span / denom > 0) {
  AD_arc_radius = span / denom;
  O = A.clone()
    .addScaledVector(adUnit, AD_arc_radius * Math.cos(alpha))
    .addScaledVector(new THREE.Vector3(0, 0, 1), -AD_arc_radius * Math.sin(alpha));
}
```

涉及公式：

```text
denom = cos(alpha) - cos(alpha + phi_c)
AD_arc_radius = span / denom
```

以及圆心构造：

```text
O = A + adUnit * (R * cos(alpha)) + Z_axis * (-R * sin(alpha))
```

该部分不是简单 three.js API 调用，而是显式几何推导，用于后续旋转生成几何结构。

1. 先按后端协议读取原始 `cameraRotation`
2. 重排到 three.js quaternion 顺序
3. 转成前端实际显示使用的 `model quaternion`
4. 叠加当前场景自己的滑条增量
5. 导出时再把 `model quaternion` 逆回 `camera quaternion`
6. 最后按后端协议写回 `[w, x, y, z]`

公式如下：

```text
Q_backend_raw = [w, x, y, z]
Q_backend_three = reorder(Q_backend_raw) -> [x, y, z, w]
Q_model = inverse(Q_backend_three)
Q_model_corrected = Q_model * Q_pitch * Q_yaw * Q_roll
Q_backend_corrected = inverse(Q_model_corrected)
Export = [w, x, y, z] from Q_backend_corrected
```

可简写为：

```text
Q_model = inverse(reorder(Q_backend))
Q_model_corrected = Q_model * Q_pitch * Q_yaw * Q_roll
Q_backend_corrected = inverse(Q_model_corrected)
```

对应实现语义：

```ts
const qBackend = new THREE.Quaternion(rotation[1], rotation[2], rotation[3], rotation[0]).normalize();
const qModel = qBackend.clone().invert();
const correctedModel = composeQuaternionFromBaseAndDeltas(qModel, pitchDelta, yawDelta, rollDelta);
const correctedBackend = correctedModel.clone().invert().normalize();
return [correctedBackend.w, correctedBackend.x, correctedBackend.y, correctedBackend.z];
```

### 7.2 Euler 场景的导出说明

若场景原始 `rotation` 不是 4 元 quaternion，而是 Euler 数据，则当前仍走 Euler 兼容链路。

这意味着：

- quaternion 场景：显示链路和导出链路现已基本对称
- Euler 场景：导出对称性仍不如 quaternion 场景清晰

这也是当前保留下来的一个后续风险点。

### 7.3 当前结论

导出阶段现在已经修复了此前最关键的不一致：

- 显示时按 `[w,x,y,z] -> [x,y,z,w] -> inverse()` 解释
- 导出时按 `inverse() -> [w,x,y,z]` 还原

因此 quaternion 场景的“显示结果”和“导出结果”现在是同一套协议下的双向变换。

## 8. 本次排查后新增的实现结论

### 8.1 多场景独立滑条已恢复

当前每个场景分别保存自己的：

- `pitch`
- `yaw`
- `roll`

并在切换场景时重新读取该场景自己的原始 `rotation` 作为 `Q_base`。

因此不会再出现“后切入的场景沿用前一个场景姿态作为基础姿态”的问题。

### 8.2 场景切换时已改为立即刷新姿态

当前在场景切换监听链路中，会立刻执行两步：

1. 先按当前场景原始 `rotation` 应用基础姿态
2. 再按当前场景保存的滑条增量叠加姿态

这修复了此前的现象：

- 切到新场景时先看到上一个场景的姿态
- 直到拖动新场景滑条后，才突然跳回当前场景自己的原始姿态

### 8.3 localStorage 恢复链路已清理

此前存在把 `modelGroup.quaternion` 直接持久化并恢复的旧逻辑，这会把前端瞬时显示态误当作基础真值，引入额外漂移。

当前已清理为：

- 不再把全局 `modelGroupQuaternion` 当作可靠恢复源
- 原始旋转以场景自己的 `rotation` 为准
- 滑条增量单独记录
- 恢复显示时重新走当前统一公式

### 8.4 精度显示链路已统一到主 quaternion 链路

此前 `updateCameraPrecisionFromProps()` 在处理 4 元 quaternion 时，曾直接按 `[x,y,z,w]` 读取，这与后端真实协议不一致。

当前已改为复用主链路构造函数，等价于：

```text
buildFrontendQuaternionFromRotation(cr).quaternion
```

这意味着当前这些功能已统一：

- 模型显示
- 滑条基准姿态
- 精度显示
- quaternion 导出

## 9. 右手系一致性检查结论

### 9.1 当前已明确正确的部分

本次排查后，可以较明确地认为以下链路已统一：

1. quaternion 主显示链路
   - 后端 `[w,x,y,z]`
   - 前端 three.js `[x,y,z,w]`
   - 固定 camera 下对 `modelGroup` 使用逆四元数

2. 多场景独立滑条链路
   - 每个场景使用自己的 `Q_base`
   - 每个场景保存自己的 delta

3. 场景切换即时姿态刷新
   - 不再残留上一个场景姿态

4. quaternion 导出链路
   - 与显示链路对称

5. precision / 调试显示链路
   - 已与主 quaternion 构造逻辑统一

### 9.2 仍可疑但暂未改动的部分

以下内容仍带有明显历史补偿或符号约定差异，但目前为了避免破坏已稳定的 quaternion 主链路，暂未继续修改：

1. `src/components/ThreeScene.vue` 中 Euler 兼容函数
   - `convertBackendEulerToFrontendEulerDeg(...)`
   - 仍存在 `-x`、`-y`、`90-z` 这类补偿

2. `src/components/ThreeScene.vue` 中 XY Rotation 的符号约定
   - `applyXYRotToPoint()` 使用 `-angle`
   - `applyXYRotInverse()` 使用 `+angle`
   - 两者互逆关系成立，但方向直觉仍需后续统一审查

### 9.3 一个容易误导排查的点

当前 `AxesHelper` 挂在 `modelGroup` 下，因此它显示的是模型局部轴，不是世界轴。

这不会导致功能错误，但在排查“世界坐标轴方向是否正确”时容易造成误判。后续如需进一步校验右手系，可考虑补一个挂在 `scene` 根节点上的世界坐标辅助轴。

## 11. 最终结论

本项目内关于模型旋转的实现，当前可以归纳为以下几条已经确认的主公式与结论。

### 11.1 quaternion 主显示公式

对于后端 quaternion 场景：

```text
Q_backend_raw = [w, x, y, z]
Q_backend_three = reorder(Q_backend_raw) -> [x, y, z, w]
Q_model = inverse(Q_backend_three)
```

这是当前最核心、也最关键的修正结果。

### 11.2 姿态滑条合成公式

```text
Q_base = buildFrontendQuaternionFromRotation(scene.rotation).quaternion
Q_target = Q_base * Q_pitch * Q_yaw * Q_roll
```

其中：

- `Q_pitch = axisAngle(X, pitch)`
- `Q_yaw = axisAngle(Y, yaw)`
- `Q_roll = axisAngle(Z, roll)`

并且这里的 `Q_base` 是“当前场景自己的原始姿态”，不再是可能跨场景漂移的共享基础值。

### 11.3 quaternion 导出公式

```text
Q_model = inverse(reorder(Q_backend))
Q_model_corrected = Q_model * Q_pitch * Q_yaw * Q_roll
Q_backend_corrected = inverse(Q_model_corrected)
Export = [w, x, y, z]
```

因此当前 quaternion 场景已经满足：

- 显示链路与导出链路对称
- 前端显示和后端存储遵循同一套协议转换

### 11.4 XY Rotation 旋转点公式

```text
θ = degree * π / 180
Q_xy = axisAngle(Z, -θ)
P' = applyQuaternion(P, Q_xy)
```

### 11.5 四元数转轴角公式

```text
angle = 2 * acos(|w|)
axis = (x, y, z) / sqrt(1 - w²)
```

### 11.6 点绕任意轴旋转公式

```text
p' = c + Q(p - c)
```

### 11.7 当前整体判断

当前排查结论可以总结为：

- 项目主旋转路径是“四元数主导”
- 最关键的问题已定位为后端 quaternion 协议实际是 `[w,x,y,z]`
- 前端由于固定 camera、实际旋转 `modelGroup`，所以显示时必须对后端 quaternion 取逆
- 多场景独立滑条与切场景即时刷新问题已修复
- quaternion 场景的显示、滑条基准、精度显示、导出已经基本统一
- 仍需后续继续审查的风险点主要集中在 Euler 兼容补偿链路和若干几何旋转函数的符号约定
