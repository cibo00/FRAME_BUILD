# 四元数相机旋转问题分析记录

## 日期：2026-04-17

## 一、问题概述

网页端 3D 场景中，从后端传入的 WiLoR 手部检测四元数应用于相机旋转后，3D 指甲模型与背景剪影图片（mask）无法对齐。

## 二、数据流梳理

```
jointstest.py (WiLoR) → testdata.json (quaternion [qx,qy,qz,qw])
    → Go 后端 (generateData.go，无变换，直接透传)
        → App.vue (cameraRotation，无变换，直接透传)
            → ThreeScene.vue (applyCameraRotationFromArray，应用旋转)
```

各环节确认：
- `generateData.go`：直接读取 `rawMeta.Quaternion` 传递，无数学变换
- `App.vue:602-604`：`meta.quaternion.length === 4` 时直接赋值，无变换
- **所有变换集中在 `ThreeScene.vue` 的 `applyCameraRotationFromArray` 函数**

## 三、涉及的坐标系

### 3.1 Python Target 坐标系（WiLoR 输出）

`build_rotation_matrix_from_axes`（jointstest.py:91-107）构建的坐标系：

| Target 轴 | 来源 | 方向 |
|-----------|------|------|
| X | OpenCV Y | 向下 |
| Y | OpenCV X | 向右 |
| Z | -OpenCV Z | 向观察者 |

注意：`compute_finger_frame`（jointstest.py:55-88）中有符号翻转：
- `dir_z = -dir_z`（取反法线方向）
- `dir_y = -dir_y`（取反对齐方向）

### 3.2 Three.js 坐标系

| 轴 | 方向 |
|----|------|
| X | 向右 |
| Y | 向上 |
| Z | 向观察者 |

### 3.3 Blender 坐标系

| 轴 | 方向 |
|----|------|
| X | 向右 |
| Y | 向内（forward） |
| Z | 向上 |

### 3.4 坐标系映射关系

**Target → Three.js 直接映射**：

| Target | Three.js |
|--------|----------|
| X (下) | -Y |
| Y (右) | X |
| Z (观) | Z |

基变换矩阵：
```
P = [[0, 1, 0],    // TJS.X = Target.Y
     [-1, 0, 0],   // TJS.Y = -Target.X
     [0, 0, 1]]    // TJS.Z = Target.Z
```

P = R_z(-90°)，对应四元数 q_basis = (√2/2, 0, 0, -√2/2) 在 (x,y,z,w) 格式下。

**已通过数值验证**：Target→Three.js 直接变换 与 Target→Blender→Three.js 经由 Blender 的变换，数学结果一致。

## 四、Blender 渲染管线参数

来自 `synthdata.py`（两个版本一致）：

- **相机位置**：(0, 0, 300) — Blender 坐标系
- **相机旋转**：Euler(0, 0, 90) — 即 Blender 中绕 Z 轴旋转 90°
- **模型旋转**：通过 quaternion 应用到模型对象
- **渲染**：黑色背景 + 白色自发光材质

**关键点**：实际 mask 图片通过深度学习模型生成，非直接 Blender 渲染。

## 五、当前代码中的已知错误

### 5.1 已修复：欧拉角独立修改（开发记录260416.md）

原始代码将四元数转为欧拉角后独立修改三个分量，这是数学错误的：
```
四元数 → Euler(XYZ) → 取反X → 取反Y → 90°-Z → 新Euler(XYZ) → 新四元数
```
欧拉角三个分量是耦合的，不能独立操作。

### 5.2 已修复：缺少取逆操作

Python 四元数代表**模型旋转**，Vue 端旋转相机需要取逆。

### 5.3 当前使用的公式

```typescript
// ThreeScene.vue:462
const qBasis = new THREE.Quaternion(0, 0, -Math.SQRT1_2, Math.SQRT1_2);
const qBasisInv = qBasis.clone().invert();
const qThreejsModel = qBasis.clone().multiply(qBackend).multiply(qBasisInv);
targetQuaternion = qThreejsModel.clone().invert();
```

数学含义：`Q_cam = inv(conj(R_z(-90°), Q_python))`

### 5.4 已验证但仍存在的偏差

用户对 Scene 5 在当前公式基础上提供了滑条修正值：
- pitch = -71.12°
- roll = 111.14°
- yaw = -35.86°

这些修正值极大（远超微调范围），说明当前公式仍存在系统性偏差。

## 六、系统性偏差排查

### 6.1 数值测试结果

在 `compute_quat.py` 中测试了 200+ 种四元数变换公式，包括：

- 直接前乘/后乘标准旋转（90°/180° 绕 X/Y/Z）
- 共轭变换（含/不含取逆）
- 完整 Blender 管线公式
- Blender 相机旋转变体（R_z(±90°) 转换到 Three.js）
- 各种组合

**结果**：所有测试公式对 Scene 5 的误差均 >7°，绝大多数 >85°。这表明可能存在以下问题之一：

1. 深度学习模型生成 mask 时使用的坐标约定与已知 Blender 管线不同
2. 四元数到相机旋转的映射需要额外步骤（如额外的坐标轴翻转）
3. WiLoR 四元数本身的计算方式（特别是 `compute_finger_frame` 中的符号翻转）影响最终结果

### 6.2 修正数据.txt 的可靠性

## 七、数学推导记录

### 7.1 共轭变换（Similarity Transform）

坐标系变换的标准方法是四元数共轭变换：

```
Q_new = q_basis × Q_old × q_basis⁻¹
```

这等价于矩阵相似变换 Q_new = P · Q_old · P^T。

### 7.2 完整理论公式

从 Python 四元数到 Three.js 相机四元数，理论推导为：

```
Q_camera = (q_basis × Q_python × q_basis⁻¹)⁻¹
```

展开含义：
1. `q_basis × Q_python × q_basis⁻¹`：将模型旋转从 Target 坐标系转换到 Three.js 坐标系
2. `(... )⁻¹`：模型旋转取逆得到等效相机旋转

### 7.3 Blender 管线公式

经由 Blender 的完整管线（理论推导）：

```
Q_camera_TJS = inv(q_cvt × (R_cam_B × Q_model_B) × q_cvt⁻¹)
```

其中：
- `q_cvt = R_x(-90°)`：Blender→Three.js 基变换
- `R_cam_B = R_z(-90°)_B`：Blender 相机旋转
- `Q_model_B = q_T2B × Q_python × q_T2B⁻¹`：Target→Blender 的模型旋转

已验证此公式与直接公式数值一致。

## 八、待解决问题

1. **正确公式未确定**：当前 R_z(-90°) 共轭变换公式有系统性偏差，需进一步调查深度学习模型的坐标约定

2. **缺少 5 个场景的统一修正数据**：需要在新公式下对所有 5 个场景采集滑条修正值，以判断偏差是否一致（如果是固定的偏移旋转，可通过额外乘法修正）

3. **深度学习渲染管线的坐标约定未知**：mask 图片由深度学习模型生成而非直接 Blender 渲染，该模型的输入坐标约定是找到正确公式的关键

## 九、建议的下一步行动

1. 确认深度学习模型生成 mask 时使用的精确坐标约定和四元数应用方式
2. 在当前公式基础上，对所有 5 个场景分别采集修正值
3. 如果 5 个场景的修正值一致（固定偏移），则在公式中增加一步修正旋转
4. 如果修正值不一致，则需要重新推导完整的坐标变换链
